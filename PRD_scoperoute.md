# PRD (v2): Fable Trigger — Triage & Monitor

**Рабочее название:** `scoperoute`
**Что это:** двухфазный инструмент, чтобы (1) заранее знать, на каких проектах Claude Fable 5 применим, и (2) в реальном времени ловить его срабатывания внутри Claude Code.
**Стек:** Python + Anthropic SDK; чтение локальных JSONL-транскриптов Claude Code.

---

## 0. Что изменилось в v2

v1 описывал per-request routing-гейт (sensitivity-классификатор, авто-роут sensitive→Opus). Уточнённая задача проще и эмпиричнее: всё, что реально собрано, уже на Opus; нужно **понять и отслеживать, где Fable вообще применим**, и не тратить силы там, где он постоянно роутится на Opus.

Поэтому инструмент строится в **две фазы**, и routing-машинерия v1 убрана — решение «где Fable» принимается по фактам из Фаз 1–2, а sensitive-работа и так на Opus.

- **Фаза 1 (начинаем здесь):** per-project триаж — изолированные сессии + проверка, триггернётся ли Fable.
- **Фаза 2:** пошаговый мониторинг триггера внутри Claude Code — на каждом шаге ловим fallback Fable→Opus.

---

## 1. Цели

- **G1.** Per-project: знать заранее, применим ли Fable на проекте (Фаза 1).
- **G2.** Runtime: на каждом шаге в Claude Code видеть, когда ход свалился с Fable на Opus (Фаза 2).
- **G3.** Обратная связь: наблюдения Фазы 2 уточняют бакеты Фазы 1.
- **G4.** Дёшево и локально; никаких «уговоров» — `code_triggered`-проекты просто уходят на Opus.

## 2. Границы (коротко)

Инструмент только **детектирует и мониторит** триггер и честно роутит на Opus там, где Fable падает. Не обходит классификатор, не тащит sensitive-работу через Fable.

---

## 3. Фаза 1 — Per-project триаж (START HERE)

**Статус:** v0 реализован (`fable_triage.py`).

**3.1 Механика.** На каждый проект — **изолированная проба-сессия** (свежий контекст, никакого переноса из других проектов). Один benign-запрос в Fable с контекстом проекта; ловим `stop_reason:"refusal"` + категорию. Server-side fallback **не** включаем — нужен сырой refusal, а не замаскированный Opus-ответ.

**3.2 Два варианта контекста.** `bare` (только код: дерево + сэмпл исходников) vs `full` (то же + CLAUDE.md + `.claude/**/*.md`). Сравнение даёт третий бакет.

**3.3 Вердикты.**
- `fable_friendly` (чисто в обоих) → кодить на Fable.
- `config_triggered` (full падает, bare чисто) → чинить конфиг (категория укажет что именно; `reasoning_extraction` = убрать из CLAUDE.md/скилла «explain your reasoning»); подтвердить `claude --safe-mode`.
- `code_triggered` (bare падает) → Opus.

**3.4 Выход.** CSV `{project, verdict, bare_tripped, bare_category, full_tripped, full_category, recommendation, error}`.

**3.5 Функциональные требования.**
- FR1. Benign-инструкция, не-security: любой refusal атрибутируется контексту, не просьбе.
- FR2. Бюджет контекста на запрос (`--max-context-chars`), чтобы прогон был дешёвым.
- FR3. Идемпотентность/resume по `project`.
- FR4. Не логировать полный prompt флагнутых запросов; категорию наружу пользователю не показывать.

**3.6 Ограничение Фазы 1 (важно).** Это **cold-context** — проверка «в покое». Не ловит триггеры, всплывающие **по ходу работы** (новые файлы попадают в контекст, tool-аутпуты, рост diff, длинные цепочки). Это закрывает Фаза 2.

## 4. Фаза 2 — Пошаговый мониторинг триггера в Claude Code

**Задача.** Во время реальной сессии Claude Code на **каждом шаге** видеть, обслужил ли ход Fable или он свалился на Opus.

**4.1 Первичный механизм — tail session JSONL.**
Claude Code пишет полный транскрипт сессии **построчно и append-only, по мере работы**, в
`~/.claude/projects/<url-encoded-project-path>/<session-uuid>.jsonl`
(на Windows — `%USERPROFILE%\.claude\`; тот же путь для CLI, VS Code/JetBrains-расширения и Desktop-интеграций). Каждая строка = один JSON-объект (user / assistant / tool), и в ассистентской записи присутствуют **выбранная модель хода и токены за ход**. Отсюда:
1. `tail -f` активного `.jsonl`;
2. на каждую новую ассистентскую запись читаем поле `model`;
3. если сессия запущена на Fable, а ход обслужен Opus → на этом шаге **сработал триггер (fallback)**;
4. эмитим алерт + пишем событие `{session, turn, model, category(если есть), tokens, ts}`.

**4.2 Поиск активного файла.** Маппинг `project path → url-encoded имя папки` под `~/.claude/projects/`; активная сессия = самый свежий по `mtime` `.jsonl` (или `session-id` из `/status`). **Субагенты** — отдельные сессии, слинкованы через `parentUuid`; мониторим и их (fallback может сработать в конкретном субагенте, не во всей сессии).

**4.3 Метрики.**
- per-session и per-project **fallback-rate** (доля ходов на Opus);
- **стрик** подряд идущих Opus-ходов — индикатор «залипания» сессии;
- разбивка по категориям (`cyber / bio / chem / distillation / reasoning_extraction`), если поле присутствует.

**4.4 Комплементарные механизмы (вторичные).**
- `/config` → переключить флаг в режим паузы/подтверждения, чтобы каждый триггер становился явным.
- Claude Code hooks (event-logger) — **если** хук отдаёт метаданные хода с моделью; проверить доступные события (иначе tail — единственный надёжный путь).
- На чистом API-пути (вне CC) — `usage.iterations` как источник обслужившей модели.

**4.5 Приватность.** Транскрипты **не шифруются** на диске и могут содержать секреты (если тул читал `.env`). Монитор реэкспортирует **только метаданные хода** (`model`, `tokens`, `category`, timestamps) — не prompt-контент, не индексировать содержимое.

**4.6 Обратная связь в Фазу 1.** Если проект прошёл cold-триаж как `fable_friendly`, но в Фазе 2 даёт устойчивый fallback-rate выше порога → **переклассифицировать в `code_triggered`** и посмотреть, не конкретная ли подпапка/тип задач триггерит.

---

## 5. Поток между фазами

Фаза 1 (статический предфильтр) → решаем, где вообще запускать Fable в CC.
Фаза 2 (runtime-монитор) → подтверждаем/опровергаем на реальной работе и уточняем бакеты.
Петля: бакеты Фазы 1 — это гипотезы, Фаза 2 их валидирует по факту.

---

## 6. Интерфейсы

- **`fable_triage.py`** (Фаза 1, готово): CLI, CSV.
- **`fable_watch.py`** (Фаза 2, к сборке): tail активного `.jsonl`, stdout-алерты + JSONL-лог событий + периодический fallback-rate.
  CLI-набросок: `fable_watch.py --project PATH` (или авто по cwd) `--session latest --alert-streak 3`.

---

## 7. Acceptance criteria

**Фаза 1** (как в v0):
- AC-P1-1. 0 `sensitive`-задач ушло на Fable.
- AC-P1-2. Корректные бакеты (сверка выборочно вручную).
- AC-P1-3. Идемпотентный resume.

**Фаза 2:**
- AC-P2-1. Детект Fable→Opus на конкретном ходу в пределах ~1 записи после появления строки в `.jsonl`.
- AC-P2-2. Корректный per-session fallback-rate (сверка с ручным просмотром транскрипта).
- AC-P2-3. Алерт на стрик ≥ N подряд Opus-ходов (залипание).
- AC-P2-4. Субагентские сессии учтены.
- AC-P2-5. В лог не попадает prompt-контент / секреты.

---

## 8. Открытые вопросы

- Точная схема ассистентской записи в `.jsonl` текущей версии CC: **где именно** поле `model` и есть ли явное поле категории/fallback — проверить `head -1 <session>.jsonl | jq`.
- Различает ли транскрипт **requested vs served** модель за ход (если да — сравнивать их напрямую; если нет — сравнивать served с известной моделью сессии).
- Есть ли hook-событие уровня turn/stop с моделью (иначе tail — единственный надёжный путь).
- Порог fallback-rate для реклассификации (§4.6).
- Доступ к Fable API / CC из региона и аккаунта.

## 9. Референсы

- Claude Code docs — структура `.claude`; сессии в `~/.claude/projects/<encoded>/<session>.jsonl`, append-only.
- Транскрипт содержит per-turn **model selection + token usage** (разборы формата: Yi Huang / lm-assist).
- Инструменты чтения JSONL: `simonw/claude-code-transcripts`, `daaain/claude-code-log`, lm-assist.
- Fable-5 fallback Cookbook (`usage.iterations`, категории) — для API-пути.
- Claude Code: `--safe-mode`, `/config` «switch models when flagged»; issue `anthropics/claude-code#67306` (`reasoning_extraction` на benign «explain your reasoning»).
